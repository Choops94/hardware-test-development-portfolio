# PCIe Protocol Reference

## Overview

PCIe (Peripheral Component Interconnect Express) is a high-speed serial communication standard used to connect hardware devices inside a computer — for example, a CPU to a GPU, NVMe drive, or network card. It replaced older parallel bus standards (PCI, AGP) by moving to a serial, point-to-point architecture that scales cleanly from low-bandwidth devices to extremely high-bandwidth ones.

**Key characteristics:**
- Serial, differential signaling (not parallel)
- Point-to-point links (not shared bus)
- Split transaction architecture (requests and completions are separate)
- Scalable lane width: ×1, ×4, ×8, ×16
- Multiple generations: Gen 1 (2.5 GT/s) through Gen 5 (32 GT/s) per lane

---

## Layered Architecture

PCIe uses a three-layer model. Each layer has a specific responsibility. All layers are conceptual — at the physical level it is all just bits on a wire, but the layered model defines which part of the hardware/logic is responsible for what.

```
┌─────────────────────────────┐
│      Transaction Layer      │  "What do you want?" — intent and operations
├─────────────────────────────┤
│        Data Link Layer      │  "Did it arrive?" — reliability and error recovery
├─────────────────────────────┤
│       Physical Layer        │  "How does it travel?" — bits on differential pairs
└─────────────────────────────┘
```

### Transaction Layer

Handles the meaning and intent of communication. Generates and receives **TLPs (Transaction Layer Packets)** which carry operations such as memory reads, memory writes, and I/O operations.

Key concept: **split transaction architecture** — a read request is sent in one direction, and the completion (carrying the data) comes back separately. This frees the bus between request and completion, allowing other devices to communicate in the meantime.

### Data Link Layer

Wraps TLPs to provide reliable delivery. Adds a **Sequence Number** and **LCRC (Link CRC)** to each TLP. Maintains a **replay buffer** — a copy of every transmitted TLP that has not yet been acknowledged.

If a TLP arrives corrupted, the receiver sends a **NAK**. The Data Link Layer pulls the original TLP from the replay buffer and retransmits it autonomously — the Transaction Layer is never involved and never knows a retry occurred.

### Physical Layer

Handles serialisation of bits onto the wire as differential pairs. Manages symbol encoding, clock recovery, equalization, and lane management. Also houses the **LTSSM (Link Training and Status State Machine)** which brings the link up from a cold start.

## Packet Structure

### TLP (Transaction Layer Packet)

The fundamental unit of PCIe communication. Generated by the Transaction Layer.

```
┌──────────────────┬─────────────────     ─┬─────────────┐
│  Header (3–4 DW) │  Payload (0–1024 DW)  │  ECRC (opt) │
└──────────────────┴────────────────     ──┴─────────────┘
```

**Key header fields:**

| Field   | Purpose |
|---------|---------|
| Type    | What kind of packet: MRd (Memory Read), MWr (Memory Write), CplD (Completion with Data), etc. |
| Tag     | Unique identifier (0–255) that matches a completion back to its original request |
| Address | Target memory address for the operation |
| Length  | Payload size in DWORDs |

### Data Link Layer Framing

The Data Link Layer wraps the TLP before transmission:

```
┌──────────────┬───────────────────────────┬──────┐
│ Seq Num (2B) │           TLP             │ LCRC │
└──────────────┴───────────────────────────┴──────┘
```

- **Sequence Number** — identifies this TLP in the replay buffer for retry purposes
- **LCRC** — recalculated fresh on every transmission; used by the receiver to detect corruption

## Worked Example: Memory Read

A complete read transaction, step by step:

1. CPU wants to read data from GPU memory
2. **Transaction Layer** generates an **MRd TLP**: Type=MRd, Tag=5, Address=0x1000, Length=4
3. **Data Link Layer** wraps it: Seq=47, calculates LCRC. Stores copy in replay buffer
4. **Physical Layer** serialises bits onto differential pairs — this is what travels on the wire
5. GPU receives it. Physical Layer deserialises. Data Link Layer checks LCRC
6. **Error detected** → GPU sends **NAK**
7. CPU Data Link Layer pulls TLP from replay buffer. Same Seq=47, new LCRC calculated. Retransmits
8. GPU receives it. LCRC valid → sends **ACK**. CPU discards Seq=47 from replay buffer
9. GPU fetches data. **Transaction Layer** generates **CplD TLP**: Type=CplD, Tag=5, with data payload
10. CPU Transaction Layer matches Tag=5 → completion delivered to original requester

**Why Tag=5 stays the same:** The Tag lives in the TLP (Transaction Layer). It tracks the logical request end-to-end. The Seq Number is a Data Link Layer concept used only for the retry mechanism — these are separate concerns at separate layers.

## Link Training and Enumeration

### The Problem

When a system powers on, neither the host nor the device knows anything about the other.
Before any TLPs can flow, the physical link must be brought up from scratch — before any OS or software is running.

### LTSSM (Link Training and Status State Machine)

The LTSSM is logic within the **PCIe controller hardware** (typically integrated into the CPU or chipset). It runs autonomously and brings up the link through a defined sequence of states:

```
Detect → Polling → Configuration → L0 (normal operation)
```

| State         | What happens |
|---------------|-------------|
| **Detect**    | Controller checks for receiver termination impedance on each lane — confirms something is physically connected |
| **Polling**   | Exchanges **TS1/TS2 training sequences** to establish bit lock (when to sample each bit) and symbol lock (where symbol boundaries are within the bit stream) |
| **Configuration** | Negotiates link width (how many lanes) and data rate (Gen 1/2/3/4/5). Always settles at the highest mutually supported combination |
| **L0**        | Normal operating state. TLPs can now flow |

**Bit lock vs symbol lock:**
- *Bit lock* — receiver synchronises its internal clock to know when to sample each incoming bit
- *Symbol lock* — receiver finds the correct alignment boundary within the encoded symbol stream (e.g. where each 10-bit group starts in 8b/10b encoding)

**Generation negotiation:** A Gen 4 host connected to a Gen 2 device will train at Gen 2. The Gen 2 device's physical layer cannot decode Gen 4 frequencies — it is a hardware capability limit, not a timing issue.

**Hot-plug sequence:** The LTSSM runs continuously in hardware. When a device is hot-plugged, the controller detects it and runs link training first. The OS is then notified, reads Configuration Space, and assigns resources. Hardware always runs before software in this sequence.

### Enumeration

Once L0 is established, the OS discovers and configures the device:

1. BIOS/OS reads **Configuration Space** from the device — a standardised 256-byte register layout at a known location that every PCIe device must expose
2. Configuration Space contains: Device ID, Vendor ID, device class, supported capabilities, Base Address Registers (BARs)
3. OS assigns memory address ranges, IRQs, and other resources
4. Device is now accessible to drivers and applications

This is why you can observe PCIe enumeration in Linux using tools like `lspci` — you are reading the Configuration Space that was discovered during this process.

## Key Terminology

| Term | Definition |
|------|-----------|
| TLP | Transaction Layer Packet — the fundamental unit of PCIe data transfer |
| MRd | Memory Read Request TLP |
| MWr | Memory Write TLP |
| CplD | Completion with Data TLP — the response to a read request |
| Tag | Unique identifier in a TLP header matching completions to requests |
| Seq Num | Data Link Layer sequence number used for replay buffer management |
| LCRC | Link CRC — error detection field added by the Data Link Layer |
| LTSSM | Link Training and Status State Machine — brings PCIe link up from cold start |
| TS1/TS2 | Training Sequences exchanged during Polling state |
| NAK | Negative Acknowledgement — triggers a TLP retry |
| ACK | Positive Acknowledgement — allows TLP to be discarded from replay buffer |
| Lane | A single differential pair (TX + RX). Links can be ×1, ×4, ×8, ×16 |
| Configuration Space | Standardised register layout exposed by every PCIe device for OS enumeration |
| GT/s | Gigatransfers per second — the raw line rate unit used for PCIe generations |

## PCIe Generations — Speed Reference

| Generation | Line Rate | Encoding | Effective Bandwidth (×16) |
|------------|-----------|----------|--------------------------|
| Gen 1 | 2.5 GT/s | 8b/10b | ~4 GB/s |
| Gen 2 | 5.0 GT/s | 8b/10b | ~8 GB/s |
| Gen 3 | 8.0 GT/s | 128b/130b | ~16 GB/s |
| Gen 4 | 16.0 GT/s | 128b/130b | ~32 GB/s |
| Gen 5 | 32.0 GT/s | 128b/130b | ~64 GB/s |

**Note on encoding overhead:** 8b/10b encodes 8 bits of data as 10 bits on the wire (20% overhead). 128b/130b encodes 128 bits as 130 bits (~1.5% overhead) — one reason Gen 3+ is more efficient per GT/s than Gen 1/2.

## Common Mistakes to Avoid

**Confusing Tag and Sequence Number** — Tag is a Transaction Layer concept tracking a logical request/completion pair across the bus. Sequence Number is a Data Link Layer concept tracking a specific TLP for retry purposes. They are separate fields with separate purposes.

**Thinking layers are separate physical things** — Transaction, Data Link, and Physical are conceptual divisions of responsibility. On the wire it is all just bits. The layering describes which hardware block is responsible for what.

**Thinking the OS runs link training** — The LTSSM runs in PCIe controller hardware before any software loads. The OS is a consumer of the result, not the initiator.

**Confusing line rate (GT/s) with bandwidth** — GT/s is the raw transfer rate before encoding overhead. Always account for encoding efficiency when calculating actual data throughput.

## Quick Reference: Linux PCIe Tools

```bash
lspci                        # List all PCIe devices
lspci -v                     # Verbose: capabilities, BARs, IRQs
lspci -vv                    # Very verbose: full Configuration Space
lspci -t                     # Tree view showing bus topology
lspci -n                     # Show vendor/device IDs numerically
sudo lspci -xxx              # Hex dump of Configuration Space

# Show a specific device (use BDF from lspci output)
lspci -v -s 01:00.0

# Monitor link status
sudo lspci -vv | grep -i "lnksta"    # Current link speed and width
sudo lspci -vv | grep -i "lnkcap"    # Maximum supported speed and width
```
